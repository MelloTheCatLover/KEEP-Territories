# KEEP Территория Побед

Командная веб-игра в стиле «захвата территорий» на гексагональной карте, в которой команды учеников 14–17 лет соревнуются, решая задания по программированию. Преподаватель выступает администратором: модерирует ответы, настраивает карту, управляет командами.

Проект — дипломная работа (ВКР). Стек: TypeScript на бэкенде и фронтенде, PostgreSQL без ORM, React-SPA. Без сторонних SaaS, без сборщиков-комбайнов.

## Что умеет платформа

**Для команд:**
- Создание команды по клику на свободный «домашний» сектор карты. Цвет команды — из палитры из 8 тонов.
- 4 действия на секторе: захват, перехват, укрепление, снятие укрепления.
- Авто-розыгрыш задания из пула при старте действия (анимированное «колесо»).
- Сброс зависшего действия (drop) — с понятным штрафом по влиянию/опыту и риском потери характеристик.
- Прокачка 5 командных характеристик (Сила / Интеллект / Выносливость / Лидерство / Удача) за уровни.
- 8 кубков по разным метрикам + общий зачёт.

**Для преподавателя (админ):**
- Генерация карты: 1–6 концентрических колец с настраиваемой сложностью на каждом.
- Модерация заявок: одобрить или отклонить с обязательным комментарием.
- Полное управление командами: ресурсы, характеристики, цвет, имя, исключение участников.
- Привязка заданий к конкретным секторам.
- Глобальные параметры (пороги опыта, максимум укрепления).

**Заданий — 85**, по темам школьной информатики (переменные, циклы, функции, структуры данных, алгоритмы, ООП, сложность, парадигмы). Все — open-question с ручной проверкой.

## Архитектура

Подробный документ: [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md).

Кратко:

```
React SPA  ⇄  Express API (Routes → Controllers → Services)  ⇄  PostgreSQL
                  │                                                  │
                  └── транзакции, SELECT FOR UPDATE                   └── CHECK / UNIQUE / partial UNIQUE
```

- Слойная backend-архитектура: тонкие маршруты, валидирующие контроллеры, сервисы как **единственный** путь к БД.
- Бизнес-правила защищены **трижды** — UI, сервис, БД.
- Производные значения (влияние, опыт, уровень, очки апгрейда, статы) **никогда не хранятся** — они вычисляются из исходных таблиц.
- Frontend разбит на feature-вертикали; общие примитивы в `shared/`.

## Локальный запуск

### Требования
- Node.js 20+ и npm
- PostgreSQL 14+

### 1. Клонирование и установка
```bash
git clone <repo-url>
cd VKR
(cd server && npm install)
(cd client && npm install)
```

### 2. База данных
Создайте пустую БД, например `vkr`:
```bash
createdb vkr
```

### 3. Конфигурация сервера
```bash
cp server/.env.example server/.env
# отредактируйте server/.env под свою БД и придумайте JWT_SECRET
```

### 4. Миграции
```bash
cd server
npm run migrate
```
Раннер идемпотентен: повторный запуск пропустит уже применённые миграции. По завершении в БД появятся 85 учебных заданий по программированию.

### 5. Запуск
В двух терминалах:
```bash
# терминал 1 — сервер на :5000
cd server && npm run dev

# терминал 2 — клиент на :5173
cd client && npm run dev
```
Откройте http://localhost:5173.

### 6. Назначение администратора
Зарегистрируйте пользователя через UI, затем в БД:
```sql
UPDATE users SET role = 'admin' WHERE email = 'you@example.com';
```
Под админом сгенерируйте карту в `/admin/map`.

### 7. (Опционально) Демо-команды
После генерации карты:
```bash
cd server && npm run seed:demo
```
Создаст 3 команды (`alpha@demo.local` / `bravo@demo.local` / `charlie@demo.local`, пароль `password123`).

## Деплой в прод (Docker)

Прод-стек: один Docker-compose, четыре сервиса. Caddy раздаёт SPA и проксирует `/api/*` в Node. Postgres закрыт внутри сети, бэкап-контейнер делает ежедневный `pg_dump`.

```
Internet ─► Caddy (:80/:443, авто-LE)
              ├── /api/*  → server:5000  → db:5432
              └── /*      → SPA static (/srv)
                                    db-backup → ./backups/db-*.sql.gz
```

### Что нужно один раз

1. **VPS** с публичным IP, Ubuntu 24.04, root-доступ по SSH.
2. **Домен** с A-записями `@` и `www` на IP VPS (порт 80/443 наружу).
3. На VPS: Docker + compose plugin + git.

```bash
ssh root@<ip>
apt update && apt install -y docker.io docker-compose-plugin git ufw
systemctl enable --now docker
ufw allow 22/tcp && ufw allow 80/tcp && ufw allow 443/tcp && ufw --force enable
```

### Первый деплой

```bash
git clone <repo-url> /opt/keep-territories
cd /opt/keep-territories
cp .env.prod.example .env
# отредактируйте .env:
#   DOMAIN=ваш-домен,www.ваш-домен
#   DB_PASSWORD=$(openssl rand -hex 24)
#   JWT_SECRET=$(openssl rand -hex 64)
docker compose --env-file .env up -d --build
```

Сервер сам прогонит миграции при старте. Caddy получит Let's Encrypt-сертификат при первом обращении.

Проверка: `curl -I https://ваш-домен/api/health` → `200 OK`.

### Повторный деплой (новая фича)

Локально пушим в git, на VPS:

```bash
cd /opt/keep-territories
git pull
docker compose --env-file .env up -d --build server web
```

`db` не пересобирается — данные в volume `pg_data` сохраняются. Миграции (`NNN_*.sql`) применятся автоматически при старте `server` (раннер идемпотентный — повторно не применяет).

### Назначение администратора в проде

```bash
docker compose exec db psql -U vkr -d vkr -c \
  "UPDATE users SET role='admin' WHERE email='you@example.com';"
```

### Бэкапы

- Контейнер `db-backup` каждый день в 03:00 локального TZ кладёт `./backups/db-YYYYMMDD-HHMM.sql.gz`.
- Ротация: файлы старше `BACKUP_RETENTION_DAYS` (по умолч. 14) удаляются автоматически.
- Ручной дамп: `./scripts/backup-db.sh`.
- Восстановление: `./scripts/restore-db.sh backups/db-YYYYMMDD-HHMM.sql.gz` (останавливает API, дропает БД, накатывает дамп, поднимает API).

Рекомендуется дополнительно синкать `./backups/` на внешнее хранилище (S3 / rsync.net / другой сервер) системным cron-ом — Docker-volume на самом VPS не выживет потерю диска.

### Локальная разработка после деплоя

Дев-флоу не меняется — Docker нужен только в проде:

```bash
cd server && npm run dev    # API на :5000
cd client && npm run dev    # SPA на :5173 (Vite читает client/.env.development)
```

Клиент в проде использует относительный `/api`, в деве — `VITE_API_BASE_URL` из `.env.development`.

## Геймплей

Карта — гексагональное поле с настраиваемым числом колец. В центре — ядро (`core`), снаружи — кольца сложности `hard`/`medium`/`easy`, в углах внешнего кольца — 6 «домашних» секторов команд.

**Старт.** Игрок без команды видит подсвеченные свободные домашние сектора → кликает → выбирает имя и цвет → создаёт команду. Команда сразу получает опыт за домашний сектор.

**Действие.** Команда кликает соседний к своей территории сектор → выбирает действие (захват / перехват / укрепление / снятие укрепления) → сервер выбирает задание из пула → клиент крутит «колесо» → команде показывается задание. Параллельно команда не может вести два действия (одна pending заявка на команду).

**Решение.** Команда решает задание и показывает преподавателю. Тот одобряет (применяется эффект) или отклоняет (заявка отменяется с комментарием). Если задание зашло в тупик — команда может сбросить сектор сама, заплатив штрафом.

**Прокачка.** За уровни команда получает «очки апгрейда» и прокачивает 5 характеристик. Drop с потерей уровня снимает случайную характеристику.

**Кубки.** 8 рейтингов (влияние, опыт, секторы, ядро, сумма характеристик, стрик без drop, перехваты, «чемпионы») + общий зачёт. Часть кубков «приватные»: на карте видно только место.

## Админ-флоу

Хаб `/admin` ведёт в 7 подстраниц:

- **Карта** — генерация (1–6 колец, сложность на каждом) и полный сброс.
- **Задания** — редактирование пула, привязка к сложности.
- **Очередь** — модерация заявок: подтвердить или отклонить с комментарием.
- **Команды** — править имя/цвет, выдать ресурсы и характеристики, исключить участника, удалить команду.
- **Привязка заданий** — какие задания доступны на конкретных секторах.
- **Кубки** — детализация рейтингов с описанием правил расчёта.
- **Настройки** — глобальные параметры игры.

Назначение администратора — только вручную через `UPDATE users SET role='admin'`.
